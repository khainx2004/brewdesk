# Môi trường chạy — thiết lập và cạm bẫy đã gặp

Cập nhật lần cuối: 20/07/2026

## Chạy nhanh

```bash
# 1. Database — Docker rootful, PHẢI có sudo
sudo docker compose up -d

# 2. Backend — PHẢI export biến môi trường trước
set -a && . ./.env && set +a
cd source/backend && ./mvnw spring-boot:run     # cổng 8080

# 3. Frontend
cd source/frontend && npm run dev               # cổng 5173
```

Đăng nhập: `admin` / `admin123` (bị bắt đổi mật khẩu lần đầu).

Backend chạy được ở terminal **hoặc** IntelliJ, không phải cả hai — xem mục
IntelliJ bên dưới.

## Cạm bẫy đã gặp — đừng mất thời gian chẩn đoán lại

**Lỗi `password authentication failed for user "brewdesk_user"`**
Gần như luôn là do quên export biến môi trường, không phải sai mật khẩu DB.
`application.yml` cố ý không đặt giá trị mặc định cho `DB_PASSWORD` và
`JWT_SECRET` để app fail ngay thay vì chạy với credential lộ trong source.
Spring Boot **không** tự đọc file `.env`.

**Exit code không phản ánh app có sống hay không.** `./mvnw spring-boot:run` in
`BUILD SUCCESS`, IntelliJ in `Process finished with exit code 0` — cả hai đều
xuất hiện kể cả khi app chết ngay lúc khởi động. Phải đọc log. Đừng dùng exit
code này trong script deploy.

**Máy chỉ có JRE thì không compile được.** Cần gói `-devel` để có `javac`:
`sudo dnf install java-25-openjdk-devel`.

**Database chạy bằng Docker rootful, không phải podman.** Tài liệu này từng ghi
nhầm là podman — podman rootless trên máy dev không có container nào cả. Daemon
`docker.service` chạy như system service nên container sống độc lập với
terminal, đóng terminal hay logout đều không sao. User `khaingh04` **không**
thuộc nhóm `docker` nên mọi lệnh `docker` đều cần `sudo`; thiếu sudo thì lỗi là
`permission denied ... unix:///var/run/docker.sock`.

**Container `brewdesk_db` mặc định `restart=no` — reboot xong DB không tự lên.**
Triệu chứng là backend báo `Connection refused ... localhost:5432`, nhìn dễ
tưởng sai cấu hình DB. Bật tự khởi động:
`sudo docker update --restart unless-stopped brewdesk_db`.
Chọn `unless-stopped` thay vì `always` để lúc chủ động `docker stop` thì nó
không tự bật lại.

**Cổng bận vì tiến trình cũ còn chạy.** `Port 8080 was already in use` (backend)
hoặc vite tự nhảy sang 5174 (frontend) gần như luôn là do một tiến trình từ phiên
trước chưa tắt, không phải lỗi cấu hình. Tìm thủ phạm:
`ss -ltnp | grep -E ':(8080|5173)'`, rồi `kill <pid>`.
Riêng vite chạy ở 5174 vẫn hoạt động bình thường — proxy `/api` khai trong
`vite.config.js` gắn với dev server chứ không gắn với số cổng.

**Cẩn thận khi dừng app bằng `pkill`.** Pattern `spring-boot:run` khớp luôn dòng
lệnh của chính script đang chạy nên nó tự giết mình. Dùng `pkill -f
"[s]pring-boot:run"` và **không** đặt chung lệnh khởi động lại trong cùng một
dòng, vì chuỗi `spring-boot:run` xuất hiện lại sẽ làm bracket trick vô hiệu.

**Spring Boot 4 dùng Jackson 3 — package `tools.jackson.*`, không phải
`com.fasterxml.jackson.*`.** Bean `ObjectMapper` Spring tạo ra là
`tools.jackson.databind.ObjectMapper`. Jackson 2 vẫn nằm trên classpath do jjwt
và springdoc kéo về nhưng không có bean nào, nên import nhầm sẽ khiến app chết
lúc khởi động với `No qualifying bean of type ObjectMapper`. Riêng annotation
(`@JsonInclude`, `@JsonProperty`...) thì Jackson 3 vẫn để ở package cũ
`com.fasterxml.jackson.annotation`.

**Sửa migration đã apply làm hỏng Flyway checksum.** Đã dính một lần: file
`V1__init.sql` rỗng bị apply nhầm, phải xóa dòng trong bảng
`flyway_schema_history` mới ghi lại schema thật được.

## IntelliJ IDEA — máy dev cài qua Flatpak

IDE là `com.jetbrains.IntelliJ-IDEA-Community` bản Flatpak, chạy trong sandbox
chỉ được cấp `filesystems=home;xdg-run/docker;xdg-run/keyring;xdg-run/podman`.
Ba hệ quả, cả ba đều từng mất thời gian chẩn đoán:

**Không thấy JDK của hệ thống.** `/usr/lib/jvm` nằm ngoài sandbox — bên trong
`/usr` là của runtime Flatpak. Danh sách SDK trống là đúng, không phải cài
thiếu. Cách sửa: **Project Structure → SDKs → + → Download JDK** (tải vào
`~/.jdks`, nằm trong home nên thấy được), hoặc copy JDK sẵn có vào home:
`mkdir -p ~/.jdks && cp -r /usr/lib/jvm/java-25-openjdk ~/.jdks/`.
Phải **copy chứ không symlink** — symlink trỏ ra ngoài sandbox thì vẫn đứt.

Flatpak đặt sẵn `JAVA_HOME=/app/jbr`, đó là JetBrains Runtime để chạy IDE. Nó
tình cờ cũng là JDK 25 nên project chạy được, nhưng nên trỏ Project SDK sang JDK
25 thật để khớp với `./mvnw` ngoài terminal.

**Không thấy `node`/`npm`.** Node cài trên host, sandbox không có. Frontend phải
chạy bằng terminal thật của Fedora, đừng dùng terminal tích hợp trong IDE.

**Run config phải khai biến môi trường thủ công.** Spring Boot không đọc `.env`,
mà nút ▶ ở lề chỉ tạo config tạm thời không lưu lại. Chuột phải nút ▶ cạnh
`main()` → **Modify Run Configuration…** → ô Environment variables, khai
`DB_PASSWORD` và `JWT_SECRET` (hai biến còn lại có mặc định trong
`application.yml`). Triệu chứng khi thiếu: log in ra `Value: "${JWT_SECRET}"`
— còn nguyên literal nghĩa là biến chưa được nạp.

Gọn hơn thì cài plugin **EnvFile** rồi trỏ tới đường dẫn tuyệt đối
`/home/khaingh04/Projects/brewDeskProject/.env`, khỏi copy secret vào IDE.

**Project mở ở `source/backend/` nên không thấy `.env`, `doc/`, `design/`** —
chúng nằm ở gốc repo, ngoài phạm vi project, Shift-Shift không tìm ra. Muốn thấy
hết: **Project Structure → Modules → `brewdesk` → Add Content Root** → trỏ tới
gốc repo.

## Sửa file rồi mà app không đổi?

`spring-boot:run` đang chạy **không** tự nạp lại file trong
`src/main/resources` (ví dụ migration mới). Phải khởi động lại tiến trình.

## Git

Repo: https://github.com/khainx2004/brewdesk (public), remote qua SSH.
Danh tính git đặt riêng trong repo, dùng email ẩn của GitHub.
Commit **không** kèm dòng Co-Authored-By.

Cố ý giữ ngoài repo: `.env` (chỉ commit `.env.example`) và
`.claude/settings.local.json`.

Về cấu hình Claude Code: `.claude/settings.json` là bản dùng chung, có trong
repo. `.claude/settings.local.json` là của riêng từng máy — nó tích lũy đường
dẫn cá nhân, session ID và có thể lẫn cả credential — bản trên máy dev từng chứa
nguyên một lệnh `psql` kèm mật khẩu database. Tuyệt đối không commit. Máy này còn
có thêm một quy tắc chặn nó ở gitignore toàn cục `~/.config/git/ignore`.
